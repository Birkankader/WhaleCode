    - Test 2: CachedPromptContext::is_valid returns false when elapsed > 300 seconds
    - Test 3: CachedPromptContext::is_valid returns false when tasks_since_cache >= 3
    - Test 4: CachedPromptContext::is_valid returns false when project_dir differs
  </behavior>
  <action>
    1. In `state.rs`, add `CachedPromptContext` struct with fields: `context: PromptContext`, `project_dir: String`, `cached_at: std::time::Instant`, `tasks_since_cache: u32`. Add constants `TTL_SECS: u64 = 300` and `MAX_TASKS: u32 = 3`. Add `is_valid(&self, project_dir: &str) -> bool` method.
    2. Add `pub cached_prompt_context: Option<CachedPromptContext>` to `AppStateInner`. Update the Default impl (Option defaults to None, so no change needed if using derive).
    3. In `commands/router.rs` `dispatch_task`, replace the direct `build_prompt_context` call with cache-aware logic:
       - Lock AppState, check if `cached_prompt_context` is_valid for the current project_dir
       - If valid: clone the context, increment tasks_since_cache, drop lock
       - If invalid: drop lock, call build_prompt_context (in spawn_blocking as currently done), then re-lock and store new CachedPromptContext
       - CRITICAL: Do NOT hold AppState lock while calling build_prompt_context (it uses ContextStore, separate state -- holding both locks risks deadlock per Pitfall 4)
    4. Write tests in state.rs for is_valid covering all four conditions.
  </action>
