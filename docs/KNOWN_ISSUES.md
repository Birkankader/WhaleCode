# Known issues

Debt ledger of deferred work. Items are tagged with a target phase and a severity:

- **cosmetic** — visible quirk, does not block user tasks
- **functional** — a real workflow is missing or awkward, users can work around it
- **architectural** — larger scope; affects the shape of later phases

Each entry is one line of what, a link to where it was last discussed, a target phase, and severity. Keep this file short — if an entry grows, move the detail into a phase spec or architecture doc and link to it from here.

Context: `docs/retrospectives/phase-2.md` bug #3. **Target:** monitor — file upstream issue if it reproduces post-Phase-4. **Severity:** cosmetic (dev-only).

### Orchestration

- **Partial run recovery is cleanup-only, not resume.** On crash, the backend marks active runs `Failed` and sweeps worktrees (`src-tauri/src/orchestration/mod.rs::recover_active_runs`). There is no "pick up where you left off." Context: `docs/retrospectives/phase-2.md` bug cluster #5. **Target:** v2.5 (headless server also wants resumable runs). **Severity:** architectural.
- **Rate-limit errors classified as `TaskFailed`, not a dedicated variant.** `classify_nonzero` (`src-tauri/src/agents/process.rs`) buckets worker-CLI non-zero exits into `ProcessCrashed` / `TaskFailed` via a stderr keyword heuristic (`/cannot|refuse|unable|failed to/i`). Upstream rate-limit messages ("rate limit exceeded", "429 too many requests") match the `TaskFailed` branch, so the retry ladder treats them as a controlled refusal — same budget as "I can't do this" — instead of a transient-backoff case that deserves a bounded retry with exponential delay. Observed secondarily during Phase 3.5's Gemini latency benchmark (some of the 243s zero-byte exits correlated with rate-limit headers visible in stderr). No user report of misclassification yet. Surfaces cleanly in Phase 4 Step 0's crash-shape diagnostic (Category B). **Target:** Phase 7 or later (alongside proper adapter-side backoff + per-provider token bucket; ties into the Phase 7 cost-aware suite for budget-cap enforcement). **Severity:** functional.
- **`classify_nonzero` is a keyword heuristic, not a structured signal.** Same function as above. Any CLI that doesn't follow the verb pattern (`cannot`/`refuse`/`unable`/`failed to`) and exits non-zero gets classified as `ProcessCrashed` regardless of intent; conversely a crashy stack trace that happens to contain "failed to" gets dropped into `TaskFailed`. The current heuristic has held up in Phase 3/3.5/4/5/6 UAT, but it's a fragile contract — a future CLI rev could swing either way. Fix would be to standardise an `exit_code == X` or `first-line prefix` convention per adapter (each adapter owns its own classifier). **Target:** no specific phase — investigate when a real misclassification is reported in production. **Severity:** architectural (latent debt; no behavioural impact today).
- **Q&A question detection is a conservative heuristic.** `detect_question` in `src-tauri/src/agents/process.rs` triggers when the last non-empty summary line ends in `?`. False-positive rate unmeasured on real adapter runs; the Skip button is the one-click escape. If the FP rate proves high after real-usage data lands, tighten by requiring a question-word start (`which` / `should` / `does` / `can` / `do` / `is`). Context: `docs/phase-5-qa-diagnostic.md` + `docs/retrospectives/phase-5.md` lesson #3 + `docs/retrospectives/phase-6.md`. **Target:** Phase 7 (only if users report noise after Phase 6 surfaces ship). **Severity:** functional.
- **Claude interactive-mode stdin injection not implemented.** Phase 5 Step 4 shipped the universal restart-with-appended-prompt path across all three adapters per Step 0's recommendation. Phase 6 Step 4's hint injection reuses the same restart-with-`extra_context` shape via `restart_with_extra`, doubling the surface that depends on the universal path. If answer-restart or hint-restart output divergence (re-run with augmented prompt producing materially different output than mid-stream continuation would have) becomes user-reported, wire Claude's interactive mode + per-subtask stdin handle. Context: `docs/phase-5-qa-diagnostic.md` response-path recommendation. **Target:** Phase 7 (also a candidate input to the Claude pause-resume diagnostic). **Severity:** functional.
- **Gemini activity-chip fidelity gap.** Phase 6 Step 2's heuristic regex matcher (verb-prefix patterns) misses any non-verb-prefix variant Gemini emits in text mode. Acceptable per Step 0 recommendation — log line still surfaces, only the chip is dropped. Tighten the regex set if Gemini-as-worker complaints surface in real usage. Context: `docs/phase-6-toolparsing-diagnostic.md`. **Target:** Phase 7 (only on user report). **Severity:** cosmetic.
- **Per-worker hint counter affordance.** Phase 6 Step 4's spec risk flag suggested "3 hints applied this session" UI to discourage hint loops (each hint loses partial worker progress). Not shipped — left for Phase 7 if real usage surfaces hint-loop pain. Context: `docs/phase-6-spec.md` Step 4 risk flags. **Target:** Phase 7 (only on user report). **Severity:** cosmetic.
- **Base-branch terminal affordance in conflict resolver.** Phase 5 Step 3's `ConflictResolverPopover` exposes per-worker Reveal-in-Finder (reusing Phase 4 Step 4's `revealWorktree`) but not an "Open terminal at base branch root" button — spec listed it but scope was satisfied by the pattern-reuse acceptance criterion. Add a new `open_terminal_at_repo` IPC + resolver button if users need faster access to the base-branch conflict state. **Target:** Phase 7. **Severity:** cosmetic (workaround: user already has a terminal open in most real workflows).

### Architecture / scope

- **Phase 7 cost-aware feature suite cluster.** A coherent sub-scope deferred from Phase 6 to keep Phase 6 free of LLM-cost user actions. The cluster covers: per-worker outcome summaries (heuristic + semantic), diff content explanations (per-file "Explain this change"), Claude pause-resume pilot (needs its own Phase 7 diagnostic step covering interactive-mode swap + tool-event format stability + Q&A-detection compatibility + feature-flag rollout cost), and a cost-dashboard foundation (per-worker / per-call / per-month / per-provider breakdowns). Wires the cost schema (Phase 2 cost tables, unused since) and introduces per-call cost preview, cumulative session tally, undo affordance, and an optional budget cap. Context: `docs/phase-6-spec.md` "What this phase does NOT include" + Q2 / Q3; `docs/retrospectives/phase-6.md` "Open debt." **Target:** Phase 7. **Severity:** architectural.
- **Multi-repo workspaces.** v2 is single-repo; mono-repo awareness was carried out of Phase 4, Phase 5, and Phase 6 scope. True multi-repo remains deferred. Context: `docs/architecture.md` decision 4; `CLAUDE.md` "7 architectural decisions." **Target:** Phase 7+ (mono-repo awareness) / v3 (true multi-repo). **Severity:** architectural.
- **Cost tracking tables unused.** Schema is in place (`src-tauri/src/storage/`), no readers or writers wired. Context: `docs/phase-2-spec.md` Step 9. **Target:** Phase 7 (wired by the cost-aware feature suite cluster). **Severity:** architectural (no behavioural impact until Phase 7).
- **Safety gate is a stub.** `is_action_safe` returns `true`. Destructive-command gating is Phase 7+. Context: `docs/phase-3-spec.md` Step 7. **Target:** Phase 7+. **Severity:** architectural.
- **Windows cancel cleanup.** Phase 3.5's cancellation fix uses `setsid` + `killpg` on Unix to kill grandchildren spawned by agent CLIs (MCP servers, tool runners). On Windows the equivalent is a Job Object assigned at spawn time (`kernel32::CreateJobObject` + `AssignProcessToJobObject`), which requires the `windows-rs` crate plus the correct flags for kill-on-close. Not wired yet — Windows cancel will still work for the direct child via `child.kill()` but grandchildren will orphan. Context: `src-tauri/src/agents/process.rs` (`install_new_process_group` / `kill_process_group`). **Target:** v2.5 (before headless mode — a server can't orphan processes). **Severity:** architectural (platform-specific, Unix shipping unaffected).

### Testing

- **No stress test for rapid submit-after-terminal.** Three bugs in Phase 2 were specific to "user acts before the event round-trip settles." No integration test covers the rapid Apply → new task → Enter path. Context: commits `28077ed`, `23e1f19`. Phase 4's Apply-summary overlay verification did not add this test, and Phase 6 surfaces (chips + thinking + hint) added new "user acts mid-flight" paths without a stress test for the fastest-finger case. The gap carries forward. **Target:** Phase 7 verification checklist. **Severity:** functional.
- **Flaky tests in the replan lifecycle family.** `replan_happy_path_accepts_replacement_and_reaches_done` failed once during Phase 3 Step 4, passed on re-run; did not recur in Step 9. During Phase 3.5 Item 5 + Phase 5 Step 5 gate runs, `replan_lineage_cap_escalates_after_chained_replans` (sibling test in the same module) also failed once under the full-suite run and passed in isolation — same flake shape. Both are timing-sensitive around the replan lifecycle (event emission vs. assertion). Workaround: run the Rust suite at `--test-threads=4` for verification passes. Phase 6 ran gates at default threading and observed no trip. If the fail rate climbs under reduced parallelism, investigate the race between `SubtaskStateChanged` dispatch and the waiter in these tests. Context: commit `6bee77c` report; `docs/retrospectives/phase-3.md` bug #6; `docs/phase-5-verification.md` note. **Target:** monitor — fix only on recurrence under `--test-threads=4`. **Severity:** functional (flaky tests mask real regressions).
- **No debug-only failure injection.** Phase 3 criteria 7, 15 (and arguably 14) could not be exercised end-to-end by hand because the app has no "force this subtask to fail" or "force ceiling exceeded" affordance — they were instead integration-verified via backend tests. Phase 4 used the Step 0 fake-agent fixtures (`src-tauri/src/agents/tests/crash_fixtures/`) to drive each crash category, which partially covers the need, but there is still no in-app runtime toggle. A small debug surface (`force_fail_next`, `force_ceiling_exceeded`, `force_question`, `force_hint_loop`) behind a dev-only flag would let later phases ship full manual verification. Phase 5 added Q&A fixtures (`question_fixtures/`) extending the pattern; Phase 6 added tool-event fixtures (`tool_event_fixtures/`) — the in-app runtime toggle is still open. Context: `docs/retrospectives/phase-3.md` lessons #6; `docs/retrospectives/phase-5.md`; `docs/retrospectives/phase-6.md`. **Target:** Phase 7 (estimate: 1 day, pays back within the phase). **Severity:** functional (test-only, but increasingly costly as the lifecycle grows).
- **No programmatic visual regression.** Three of Phase 3's six closeout bugs (title squeeze, running/done height, empty LogBlock black hole) were only caught by a human watching the card transition. Phase 4 repeated the pattern — three of its closeout fixes (expand height, z-index, diff context) were user-reported rather than test-caught. Phase 5 and Phase 6 had none of that — but only because each Step 0-style spike ahead of time let us land each step in one commit without visual surprises, not because the coverage improved. Unit tests still cover logic, not pixels. Text-only "visual observations" continue in `docs/retrospectives/phase-N-visuals/` but scale poorly. Context: `docs/retrospectives/phase-3.md` lessons #4; `docs/retrospectives/phase-4.md` bug cluster #1; `docs/retrospectives/phase-5.md` lesson #6; `docs/retrospectives/phase-6.md`. **Target:** Phase 7 pilot (Playwright + `@playwright/experimental-ct-react` vs. Storybook visual-test-runner). **Severity:** functional (process, not code).
